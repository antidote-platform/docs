This page is a tour of how Antidote Cloud is put together. Read it
once before diving into the API reference. Most other docs assume
this vocabulary.

## 1. System topology

```
┌────────────┐     HTTPS / WebSocket     ┌──────────────┐    work queues       ┌────────────────┐
│  Web UI    │ <───────────────────────> │  Antidote    │ <-----------------> │ Scanning       │
│ (browser)  │                           │   API        │                     │ workers        │
└────────────┘                           └──────┬───────┘                     └──────┬─────────┘
                                                │                                    │
                                          ┌─────▼─────┐                       ┌──────▼─────┐
                                          │ Database  │                       │  Message   │
                                          │           │                       │  broker    │
                                          └─────┬─────┘                       └──────┬─────┘
                                                │                                    │
                                                │        Healing proxy               │
                                          ┌─────▼─────┐    HTTP             ┌────────▼────────┐
                                          │  Dataset  │ <------------------ │  Healing        │
                                          │  storage  │                     │  service        │
                                          └───────────┘                     └─────────────────┘
```

| Component             | Responsibilities                                                                                                                                                |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Web UI**            | The React SPA. Dataset tables, scan controls, latent-space scatter, compliance modals, support forms.                                                           |
| **Antidote API**      | REST and WebSocket surface. Dataset and scan CRUD, uploads (zip / folder / text / repo), compliance, support, healing proxy, scan orchestration.                |
| **Scanning workers**  | Run the heavy pipelines: `mislabel_2d`, `mislabel_3d`, `mislabel_broad` (CLIP/OOD), `poisoning`, `text_analysis`. Workers share one image; the queue picks the job. |
| **Healing service**   | Builds healed dataset archives. Reachable through `GET /api/datasets/{id}/cure`.                                                                                |
| **Database**          | Datasets, scans, results, action logs, compliance and support records, users and roles.                                                                         |
| **Message broker**    | Queues scan tasks and powers the `/ws/api/scans/{id}` progress stream.                                                                                          |
| **Dataset storage**   | Where uploaded datasets, derived files, and healed archives live. Backed by managed object storage in the cloud.                                                |

## 2. Request lifecycle

A typical scan flows through every layer of the stack:

<Steps>
  <Step title="Ingest">
    `POST /api/uploads/dataset/images` accepts a zip or folder upload,
    writes the contents to dataset storage, derives metadata, and
    upserts a `Dataset` row. The text path
    (`/api/uploads/dataset/text`) ingests documents and immediately
    queues a `text_analysis` scan.
  </Step>

  <Step title="Enqueue">
    `POST /api/datasets/{id}/scan` normalizes the scan-type alias,
    persists a `Scan` row, and pushes a task onto the matching queue.
  </Step>

  <Step title="Execute">
    A worker pulls the task. The engine reads the dataset off
    storage, runs its pipeline, and emits `Result` rows with `flags`,
    `score`, latent embeddings, and any engine-specific `extra`
    fields.
  </Step>

  <Step title="Stream progress">
    The engine periodically publishes a JSON progress event.
    `/ws/api/scans/{scan_id}` subscribes to those events and forwards
    each one to the browser, driving the progress meter.
  </Step>

  <Step title="Finalize">
    The lifecycle service updates `Scan.status`, computes
    `duration_seconds`, updates the parent dataset's severity counts,
    and (if configured) POSTs to your `webhook_on_complete` or
    `webhook_on_failure` URL.
  </Step>

  <Step title="Read back">
    The UI hydrates the dataset detail panel via
    `GET /api/scans/{scan_id}` plus `/api/scans/{scan_id}/results`,
    previews via `/results/{result_id}/preview`, and overlays latent
    coordinates from `/representation`.
  </Step>
</Steps>

## 3. Engines

| Engine             | What it finds                                                                                            |
| ------------------ | -------------------------------------------------------------------------------------------------------- |
| `mislabel_2d`      | Wrong labels in 2D image folders. Embeddings plus cross-validated LightGBM with multi-signal voting.     |
| `mislabel_3d`      | Wrong labels in medical / NIfTI volumes. Multi-slice windowing on top of the 2D pipeline.                |
| `mislabel_broad`   | CLIP-based OOD detection plus multi-issue tagging. Useful when classes are coarse or open-ended.         |
| `poisoning`        | Backdoor and poisoned-sample detection blending structural metrics with semantic topic drift.            |
| `text_analysis`    | Document scanning: secrets, prompt injection, lexical outliers, topic drift.                             |
| `representation`   | Helper. Computes 2D scatter coordinates for the latent-space view when engines do not emit any.          |

Each pipeline enriches `Result.extra` and `Result.flags` with metrics
the UI reads directly (`issueScore`, `issueRatio`,
`predicted_mismatch`, `latent`, ...). The API reference documents the
exact field shapes.

## 4. Workers and progress

The dispatcher inspects the stored scan type (always a string) and
routes to one of:

| Queue            | Use                                                                 |
| ---------------- | ------------------------------------------------------------------- |
| `mislabel_fine`  | 2D mislabel, and 3D when image specificity is `fine`.               |
| `mislabel_broad` | CLIP / broad OOD. Also when image specificity is `broad`.           |
| `poisoning`      | Poisoning detector plus follow-up heuristics.                       |
| `text_analysis`  | Text-dataset analysis.                                              |

The lifecycle service owns status transitions, dataset rollups, and
webhook delivery. The progress publisher normalizes engine callbacks
and pushes them to the broker; the WebSocket route is a thin
subscriber.

## 5. Healing workflow

- `GET /api/datasets/{id}/cure` returns a healed copy of the dataset.
- Three independent toggles: relabel mislabeled files, drop outliers,
  drop poisoned rows.
- Every request writes a `dataset_action_logs` entry, so the audit
  trail captures who exported what and when.

## 6. Compliance and support

- Compliance reports are produced in JSON, CSV, or PDF, scoped to a
  single dataset or to the whole workspace.
- `dataset_action_logs` captures every meaningful action (uploads,
  scans, cures, compliance exports), so the UI can render a complete
  audit trail and reports can quote historical activity.
- Bug reports and feature requests submitted from the UI flow into
  the same support tables, with optional attachments.

## 7. Web UI data flow

```
datasetApi (services/api.ts)
        │ fetch JSON / WebSocket
        ▼
hooks/useDetailsTableData ─> table row transformers & metrics
        │
        ├─ useDatasetOverview / useScanOverview (stats, progress)
        └─ useTableData (issue-threshold filtering)
```

- **DetailsTable** renders dataset findings, preview modals, and
  poisoning follow-up actions while honoring the threshold slider.
- **RepresentationModal** calls `/api/scans/{id}/representation` and
  filters classes and issue types with a scatter plot.
- **Actions** centralizes scan, cure, delete, and compliance-report
  triggers.

## 8. Data model

| Table                              | Purpose                                                                                                          |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `datasets`                         | One row per dataset directory. Severity, vulnerability counts, webhook settings, repo metadata, classes, `meta`. |
| `scans`                            | One row per scan request. `scan_type` as a plain string, params, normalized status / progress, task IDs.         |
| `results`                          | Per-asset findings with scores, flags, latent embeddings, preview references, engine extras.                     |
| `dataset_action_logs`              | Append-only audit trail. Uploads, scans, cures, compliance exports.                                              |
| `bug_reports` / `feature_requests` | Support submissions plus attachment metadata.                                                                    |
| `users` / `roles` / `user_roles`   | Authentication and authorization.                                                                                |
